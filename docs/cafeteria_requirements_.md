**University Cafeteria Ordering System**

*Software Requirements Specification (SRS)*


**56 Functional Requirements \| 24 Edge Cases \| 32 Non-Functional Requirements**



**1. Functional Requirements**

56 requirements across 8 categories. Edge case requirements are highlighted in yellow and also consolidated in Section 6.

| **FR30**                      | Payment         | System prevents double-charging by ensuring each payment attempt is processed with a unique idempotency key at the gateway level                                                                       | **Must-Have**   | **YES**       | Edge case: network retry causes duplicate charge. | **FR31**                      | Payment         | Meal Plan payment is rejected if the user\'s Meal Plan balance is insufficient; user is shown current balance and shortfall                                                                            | **Must-Have**   | **YES**       | Edge case: insufficient Meal Plan balance. | **FR32**                      | Payment         | Wallet balance is deducted atomically on order confirmation; if deduction fails, order is not confirmed                                                                                                | **Must-Have**   | **YES**       | Edge case: race condition on wallet debit --- use DB transaction with SELECT FOR UPDATE.                                                     |
| **FR33**                      | Payment         | System sends order confirmation notification (in-app and email) within 30 seconds of successful payment                                                                                                | **Should-Have** | **No**        | Async notification job. | **FR35**                      | Order Lifecycle | Staff updates order status forward through the state machine (CONFIRMED → PREPARING → READY → COLLECTED); backward transitions are not permitted                                                       | **Must-Have**   | **No**        | Role guard: only Staff and Admin can advance order status.                                                                                   |
| **FR36**                      | Order Lifecycle | User tracks their active order in real time with status updates pushed via server-sent events or polling every 15 seconds                                                                              | **Must-Have**   | **No**        | Real-time tracking. | **FR39**                      | Order Lifecycle | Admin or Staff can cancel any order at any status (except COMPLETED) with a mandatory cancellation reason code                                                                                         | **Must-Have**   | **No**        | Reason codes: Out of stock, Customer request, Staff error, System error, Suspicious order.                                                   |

| **FR41**                      | Order Lifecycle | System detects stock inconsistency (ordered quantity exceeds available stock) discovered after confirmation and immediately notifies admin; order is held pending admin resolution                     | **Must-Have**   | **YES**       | Edge case: stock count drift between lock acquisition and physical preparation. **4. Order State Machine Reference**
